论文名称：Speculator：A Tool to Analyze Speculative Execution Attacks and Mitigations

论文类型：期刊

发表位置：Proceedings of the 35th Annual Computer Security Applications Conference

发表年份：2019

# 摘要

现状：新的推测执行攻击基本上每个月都会出现新的，同时仍旧存在很多未被发现的微体系结构攻击 Speculator：一个新的工具，用于审查新的微体系结构攻击和相应的缓解方法（GDB of speculative execution） 通过利用一些可被观察到的性能计数器，speculator可以了解代码片段或者复杂攻击常见在推测执行过程中的微体系结构行为

# 介绍

目前推测式攻击和防御的需求：

为了开发新的攻击，通常会需要分析调试一些PoC代码的片段，此时就需要一个类似于推测执行攻击的工具，能够直接检查微体系结构状态的变化，从而判断攻击是否成功

当测试防御策略的有效性时，目前的选择是执行攻击的PoC代码或者选择相信一些提供的CPU flags或者内核配置。此时也需要一个更细粒度的工具能够检查微体系结构状态的变化，从而判断防御策略的有效性

推测执行攻击的步骤

  1. 准备侧信道：执行一些操作用于增加攻击成功的机会。例如对于prime-and-probe的攻击，需要实现刷新cache。
  2. 准备侧信道攻击：执行一些代码，引导受害者进行推测执行
  3. 开始推测执行：CPU执行一条指令，并且该指令的输出将会决定下一条指令的执行，例如条件分支。
  4. 推测执行，通过侧信道发送数据：推测执行某些指令，修改微体系结构状态（侧信道）
  5. 通过侧信道接收数据：CPU执行某些指令，将微体系结构状态的变化转换为数据。

推测执行攻击可以跨域的特权边界：

  1. kernel vs. user-mode code
  2. hardware enclave(SGX) vs. user-mode or kernel-mode code
  3. sandboxed code in the same process, (JavaScript JIT code)
  4. processes-to-process boundary
  5. remote node to local node boundary

Speculator：通过性能计数器记录或者推断微体系结构行为，支持增量分析(在代码片段(code snippet)上的微体系结构状态的演化)，在Intel和AMD cpu上运行，支持并发执行(在SMT环境中两个线程的交互)

文章工作:

  1. 提出了一种基于性能计数器的方法和工具，speculator，用于分析攻击和缓解策略
  2. 利用speculator验证了RAS的大小，嵌套的推测执行的有效性，推测执行不会跨越系统调用，clflush在推测执行过程中无效等。同时也测量了对于间接分支，间接控制流转移和store-to-load前递导致的推测窗口的大小。最后测试了页面权限，内存保护扩展和特殊指令（例如lfence）对推测执行的影响
  3. 给出了一些利用speculator分析的攻击和缓解示例

# Performance Monitor Capabilities

现在处理器提供了一组寄存器（固定的和可编程的），用于展示执行的各个方面的信息。这些寄存器或者计数器用于记录一些和微体系结构状态变化相关的时间或者持续时间，例如提交指令数，转移预测错误次数等。

使用性能计数器的流程：

  1. setup阶段，开发人员选择一些事件由可编程的计数器测量
  2. start和stop，测量过程可以通过编程控制开始和结束，从而准确的控制事件记录的过程
  3. access阶段，获取计数器的值
  4. setup，start，stop通常需要supervisor模式（ring 0），access在用户模式即可

# Objectives of Speculator (目的)

主要目的：准确测量代码片段中推测部分相关的微体系结构状态属性。精度是指工具能够将代码片段所引起的微体系结构状态变化和工具本身以及系统其余部分隔离开的程度（外界的影响）。

Speculator可观察到的部分信息：

  1. 代码中那些部分是被推测执行的
  2. 引发推测执行和结束推测执行的原因
  3. 影响推测执行指令数量/指令窗口的参数
  4. 特定的指令如何影响推测执行的行为
  5. 那些安全边界可以有效的阻止推测执行
  6. CPU在同一个体系结构内以及跨体系结构和供应商的行为如何保持一致（consistency）

Speculator的另一个目标：为代码片段的生成和操作（manipulation）提供一个工具。Speculator可以在推测执行期间检查单个代码片段或者代码片段组，从而使得用户只需要专注于特定用例相关的指令组合。

# Speculator的设计和实现：

三个主要部件：预处理单元，运行时单元（Monitor），后期处理单元

预处理单元：将提供的输入编译成合适的执行格式，从而引入性能监视器接口所需要的代码/工具，以便能够观察所选的计数器的值。

  1. 输入：一段C或者汇编代码
  2. 处理：将预先编译的json文件中的指令添加到输入代码的特定位置（在source template中定义的位置中）

Runtime 单元：主要由speculator monitor负责初始化和读取生成代码中使用到的计数器

  1. 生成的代码片段和monitor会放在不同的核上执行，以减少相互之间的干扰
  2. 当Monitor建立好环境后，就会加载并在另一个进程中执行snippet，直到结束
  3. 代码片段的开头和结尾会对计数器进行reset，start和stop的操作

后期处理单元：

  1. 当代码片段执行完成后，OS通知monitor。之后Monitor将获取这些计数器的值，将结果写入文件中
  2. 在多次运行之后，统计收集所有结果，并且去除其中的可疑点

为了自动生成测试样例，speculator为用户提供了一系列的模板，能够用于重现各种触发推测执行的方式，包括BTI（branch target injection），spectre v1， RAS等。

# 推测执行标记（markers）：各种可被使用的计数器

推测执行标记是指某些特殊的指令或者序列，他们在未提交之前就可以被性能计数器检测到。通常情况下，这些标记会引发一些微体系结构事件，从而可以被计数器捕获到。例如计数器可以测量某些类型的指令的发射/执行的数量，同时不需要指令一定会被提交。 通过在代码片段中增加这样的标记，Speculator通过检测性能计数将可以判断CPU是否执行了该代码片段。 Intel处理器中一些可用的标记： UOPS_EXECUTED.CORE/THREAD ：通过减去最终提交的指令数量，就可以直到被推测执行但是未提交的指令的数量。该计数器不计数不需要执行的指令，例如NOP，但是不包括FNOP（not understand） UOPS_ISSUED.SINGLE_MUL ：当使用了XMM寄存器的单精度浮点指令发射时，进行计数。在最近的CPU中会删除了，例如Skylake UOPS_ISSUED.SLOW_LEA ：计数带有三个源操作数的lea指令。（clflush也被认为是该类型的指令） LD_BLOCKS.STORE_FORWARD ：当store前递的结果失败后，计数器+1 AMD Zen可用的计数器 DIV_OP_COUNT ：被执行的div指令的数量 NUMBER_ OF_ MOVE_ ELIMINATION_ AND_ SCALAR_ OP_OPTIMIZATION： 类似于LD_BLOCKS.STORE_FORWARD，但是相对而言统计的是这种指令序列出现的情况，他统计了move elimination成功的次数。

# 利用Speculator分析微体系结构状态的变化

Return Stack Buffer Size ：RSB的大小通常已知，因此可以用于衡量speculator的准确性

设计：snippet代码片段包含在call和ret之间，在代码中调用filler函数，该函数会增加嵌套调用的层数。每一层call都会在RSB中增加一个表项，如果表项数使用完了，最外层的call指令将找不到对应的ret指令，导致CPU无法执行实现设置的标记

# 可执行的页权限：executable page permission

问题：在推测执行过程中，是否可以推测执行不具有执行权限的页内的指令 实现：在推测执行过程中（branch mis-prediction），增加控制流转移指令跳转到non-executable的内存区域。在实验中，首先保证该内存区域中要被使用的数据会在L2 cache中，同时地址也在TLB中 结论：研究的所有架构中，页的可执行权限检查都会被满足，即如果该页面是不可执行的，则该页面中的指令也不会被推测执行。

# 内存保护扩展（MPX 指令系统扩展）

MPX的功能：efficiently keeping track of bounds information associated with pointers and corresponding spatial memory checks before dereferencing pointers 。如果边界检查失败，会触发一个#BR异常，CPU自陷入kernel

实验：测量在边界检查分支之后有多少代码会被推测执行

结果：MPX的FNOP指令在增加到22个之后，执行的指令数量不在发生变化

Speculator : A Microscope for Spectre-like Attacks

Speculator：A Tool to Analyze Speculative Execution Attacks and Mitigations

自从2018年初发现Spectre and Meltdown以来，我在IBM Research – Zurich的团队一直非常关注于瞬态执行攻击的研究。 在旅途中，我们很快意识到，为了研究瞬态执行攻击，我们首先需要观察并理解实际的瞬态执行：中央处理器（CPU）到底在做什么？

然而，事实证明，观察CPU在瞬时执行方面的行为绝非易事。 每当CPU无法继续运行且必须停止并等待时，就会执行瞬时执行。

让我用一个类比来解释。 想象一下，在咖啡店里，几个常规客户（用户）不断回来并订购相同样式的咖啡（程序）。 为了节省时间，咖啡师（我们的CPU）一看到他们来，为了节省时间，他甚至在下订单之前就开始准备他们的常规订单。 假设预测正确，通常可以为客户节省时间。 但是，可能有一天客户决定尝试其他方法，在这种情况下，必须丢弃咖啡师准备的咖啡。 这与CPU有何关系？ 这样的想法是，客户看不到什么时候被丢弃，而是仅收到正确的新订单，因为准备工作发生在柜台后面（我们的微体系结构）。 本质上，客户只能看到顶部的柜台交易（我们的架构）。

在计算机上，CPU基本上会“猜测”它尚不知道的内容。 如果猜测正确，我们可以安全地提高性能。 如果不是，则CPU必须回溯，撤消所有已发生的操作并重新开始。 显然，撤消必须彻底且完整，否则应用程序将产生错误的结果。 碰巧的是，尽管撤消保留了我们所有应用程序的正确性，但仍然留下了痕迹，熟练的攻击者可以利用这些痕迹来发挥自己的优势。 而且，熟练的攻击者可能会导致CPU做出错误的猜测，从而迫使CPU留下对攻击者更有用的跟踪。 这就是《幽灵与危机》教给我们的。

不幸的是，很难精确地描述和量化这些痕迹的真正含义以及在何种条件下将其留下。 由于该信息是各个CPU制造商的专有信息，因此没有手册描述瞬态执行的详细信息。 由于发生了所有网络攻击，我们现在了解的更多了，尽管我们还远远没有看到全部。 造成困难的是，甚至没有工具或方法来进行此类调查。 这就像试图在没有显微镜的情况下研究微生物学一样。

事实证明，我们只需要建立一个显微镜。 为了构建它，我们开始研究CPU本身提供给开发人员的所有信息。 我们立即专注于所有现代CPU都提供的Performance Monitor Counters接口。 实际上，此接口是专为不同目的而设计的，即研究软件性能的各种特征。 我们的第一个主要贡献是，我们能够确定如何可靠地使用性能计数器来理解投机执行。 尤其是，我们能够识别出这些计数器中的哪些可以用作代码的推测执行片段中的标记。 标记指的是这些性能计数器可检测到的任何一条指令或一系列指令，即使该指令最终作为错误的“猜测”的一部分被CPU执行，从而随后被撤消。
