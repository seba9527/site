论文名称：FANS: Fuzzing Android Native System Services via Automated Interface Analysis

关键词：安卓系统服务 模糊测试

论文类型：期刊

发表位置：usenix

发表年份：2020

# 详细阅读

# 摘要

现状：安卓原生系统服务给用户应用提供了必要的支持和基本的函数功能，找到这里面的漏洞对分析安卓的安全非常重要。 Fuzz 是非常好的一个方法，但是会面临很多问题。

  1. 这些服务是通过 binder 调用
  2. fuzzer 必须生成接口相关的测试样例
  3. 测试样例必须满足语法规则

对于这些问题，文章提出了一种自动化的基于生成的 fuzz 方法。主要功能有

  1. 收集目标系统中的所有服务接口，发现多层嵌套的接口，并进行测试
  2. 自动化从目标接口的 AST 中解析接口的数据模型，包括可行的事务代码，事务数据中的变量名称和类型
  3. 通过变量名称和类型的知识，判断事务中的变量依赖关系；通过生成和使用关系推断接口的依赖性
  4. 使用接口模型和依赖性知识，生成具有有效格式和语义的事务序列，以测试目标服务的接口

同时发现了30多个漏洞和138个 Java 异常。

# 介绍

首先说明了安卓原始系统服务的漏洞百出，但是没有很多工作去分析安卓原生系统服务中的问题，介绍了两种自动化的 fuzz 手段，说明其不完整和不准确性，或者是需要反射来获取变量类型，均不能用来恢复输入的模型。

在安卓中，系统服务被注册到 Service Manager 中，用户态应用可以通过查询来获取目标服务的接口，然后通过 RPC 调用各种不同的事务，IBinder::transact(code,data,reply,flags) ，参数包含目标事务的代码和序列化的数据，然后通过这种方法测试所有的系统服务。这里会面临三个挑战。

  1. 多层接口的识别。除了在 Service Manager 注册过的接口，有很多多层嵌套的接口，还要考虑到很多编译时动态生成的代码，比如 Android Interface Definition Language (AIDL) 。
  2. 接口模型的解析。
  3. 生成语义正确的输入。

主要贡献

  1. 我们系统地研究了 Android 本机系统服务中的接口与未挖掘的更深层次的接口之间的依赖关系。
  2. 我们提出了一种从 AST 中自动提取输入接口模型和语义的解决方案。 此方法可以应用于其他基于接口的程序。
  3. 我们提出了一种解决方案，可以通过利用不同事务中的序列化和反序列化对中的变量名和类型知识来推断事务间的依赖性。
  4. 我们实现了 FANS 的原型，以系统地模糊 Android 本机系统服务，并且发现了30个独特的本机漏洞和138个独特的 Java 异常。

# Android系统服务的系统化

安卓里的服务可以分为2部分，一个是 Java 服务，是由 Java 实现的， 一个是原生服务，基本是由 C++ 实现的。从另一个角度看，可以分为正常域、服务商域、硬件域。正常域中的服务是 Android Open Source Project (AOSP) 中的，其他的是和服务商或者硬件相关的。

# 研究范围

文章主要讨论在正确域中的系统服务，主要是在 service manager 中注册过的。同时，由于所有 Android 系统服务在通信和接口实现方面都共享相同的体系结构，因此本文提出的方案也可以应用于其他类型的服务。

# 设计

主要讨论这个 Fans 是如何设计的。

  1. 以 RPC 为中心的测试，减少难度，并且尽可能提高可靠性，因为传统的注入攻击会生成任意事件，在实际的过程中通常不可能。
  2. 基于生成的 Fuzz ，不采用基于变异的 Fuzz ，因为基于变异的数据生成通常会生成很多无格式意义的数据，代码覆盖率较低。
  3. 从代码中学习输入模型，因为采用基于生成的模糊测试，所以需要根据输入模型，但是现有的 Fuzz 的数据都是通过人为努力，提取出来的数据，对安卓服务不可用。还有的一些方法从现有的事务中提取输入，但是总体上是不完整和不准确的。最后考虑到，输入模型知识将被隐藏在源代码中，所以选择分析 Android 源代码来自动化恢复输入模型。

整个系统分为4部分，接口收集器负责收集整个系统的顶层接口和多层嵌套接口；然后接口模型解析器负责解析输入和输出格式，以及变量语义、名字和类型等信息，同时还收集一些结构体的信息；然后依赖推测器推测接口间、接口内部的变量依赖。最后基于上面的信息，模糊测试器随机生成事务并调用相关的接口，来测试系统服务，该模块还有一个管理器负责同步主机和手机间的数据。

# 接口收集器

顶层或多层的接口都有 onTransact 方法来分发事务，利用这一特性，检查了在AOSP编译命令中作为源出现的每个C / C ++文件，这样就能收集到由AIDL动态生成的接口。

# 实现

实现的话，并没有采用 AFL ，因为转换到安卓要话很多的工作；其次测试大量的服务需要一个一个编译；最后 AFL 不擅长测试这种基于服务的应用。

  1. 接口收集器，首先编译 ASOP 代码库，记录编译命令，然后找出特性。
  2. 接口模型解析器，在 AST 上做一个近似切片，仅保留与输入和输出变量相关的语句，而忽略其他语句。
  3. 依赖推测器，给定使用 JSON 描述的接口模型，将遍历该模型并进行接口依赖性推断
  4. 模糊测试引擎，实现了一个简单的模糊器管理器，以便在多部电话上运行，并在主机和智能手机之间同步数据。由于某些 Android 本机系统服务在接收 RPC 请求时会检查调用方的权限，因此模糊检查程序将以root特权执行。同时采用异步的 RPC 。为了分析触发的崩溃，我们使用 Android 的内置 logcat 工具进行日志记录。

# 评估

评估需要评估3个问题。

  1. 能找到多少个接口，接口间的关系是什么。
  2. 所提取的接口模型是什么样的，模型是否完整和与精确。
  3. FANS 能多高效得找到这些服务接口。

# 讨论

讨论 FANS 的局限性和进一步工作。

# 接口模型的精度

一是接口模型的精度仍需确认，比如对列表的循环遍历操作，循环大小不确定。二是开发人员在某处定义了语义类型，他也可能会使用原始类型而不是类型别名，因此无法获得更多的语义类型，这将影响变量依赖的生成。三是得到的依赖关系可能不完整，因为在事务调用之间可能存在特定的顺序，服务可以看作是状态机，但是，在测试的过程中，如果我们始终遵循指定的顺序，则可能会错过一些漏洞。这说明服务状态机处理不完整，而引起一些漏洞。

# 覆盖率指导下的检测

由于系统服务是状态敏感的，因此其覆盖率可能会受到前面生成的输入或其他应用调用的影响。

# 提高fuzz的效率

由于某些 Android 系统服务是守护程序运行的，或者可能会检查调用者的权限。以 root 用户身份进行测试会导致一些问题。发现即使在启动Fuzzer 10分钟之后，手机也可以进入恢复模式，需要手动刷新手机，这极大地影响了FANS的效率。

# 基于接口的模糊测试

在 Android 9 中，主要存在三种位于不同域中的服务：普通域，供应商域和硬件域，本文主要关注普通域中的本机系统服务。但是，这三种服务在通信和接口实现方面共享相同的体系结构。因此可以轻松地将该方法转移到其他服务中。

此外，还存在一些不属于上述部分的类似接口，即定制接口。 这些接口旨在由应用程序实现和实例化，并由客户端传递到服务器端，文章没有实现的原因是需要手动实例化这些接口。

# 相关工作

# 进程间通信 IPC 和 服务安全

指出前人的工作是基于 intent 的安全分析，关于 binder 这方面的工作不足。事实上 binder 才是安卓系统中的实际安全边界，有去分析 AIDL 文件的，有分析输入验证漏洞和访问控制不一致的，还有从 IPC 流量中试图理解输入模型和事务依赖关系的，但是这些方法缺乏完整性和准确性。

# Fuzzing for Structured Input

如何去 fuzz 一个输入的结构体是关键的问题之一。通常有两种方法。

  1. 基于生成的 fuzzer ，采用模板和预先定义的语法。比如 Peach ， 这些方法需要人工参与且拓展性较差。
  2. 基于突变的 fuzzer ，突变现有的测试样例，生成一个没有规则的新样例，通过动态污点分析找到输入的共同特点，也有结合符号执行进行模糊测试的。

# 文章综述

该文章提出了一种 fuzz 安卓原生系统服务的方法。能自动化的生成事务，并调用相应的接口。 fans 能自动化处理接口间的复杂依赖。这种依赖性体现在三方面：变量模式、类型别名和变量依赖性。最后发现了30多个漏洞，138个 Java 异常，这些东西要 fuzz 到，必须要有一个精确的接口模型。
